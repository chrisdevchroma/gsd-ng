# Thinking Config Source Analysis — Claude Code v2.1.63

Reverse-engineering of the thinking parameter construction in Claude Code v2.1.63 (`@anthropic-ai/claude-code@2.1.63`, published Feb 28, 2026). Source obtained via `npm pack` and analyzed from the minified bundle `cli.js` (12,829 lines).

**Research date:** 2026-04-13
**Methodology:** Static analysis of minified JavaScript bundle. Deobfuscation by pattern-matching minified variable names against their usage context.

---

## Key finding

**v2.1.63 sends `thinking: { type: "adaptive" }` to the Anthropic API for Opus 4.6 by default.** The `CLAUDE_CODE_DISABLE_ADAPTIVE_THINKING` environment variable already exists in this version and functions correctly. There is no version of Claude Code post-Opus-4.6 (Feb 9, 2026) that avoids adaptive thinking without explicitly setting this env var.

## Thinking parameter construction

### All `thinking` object literals in code paths (excluding documentation strings)

```bash
$ grep -oP 'thinking:\{[^}]+\}' cli.js | sort -u
thinking:{type:"enabled",budget_tokens:zk8}

$ grep -oP 'thinkingConfig:\{[^}]+\}' cli.js | sort -u
thinkingConfig:{type:"disabled"}

$ grep -oP 'var zk8=\d+' cli.js
var zk8=1024
```

The only `thinking` literal in code is `{type:"enabled",budget_tokens:1024}`, used for `countTokens` API calls (not main conversation). The actual thinking config for conversation is built dynamically.

### The `thinkingConfig` field is passed through `options`

The main conversation loop passes `thinkingConfig` through the options chain:

```javascript
// Subagents, compaction, and utilities hardcode disabled:
thinkingConfig: { type: "disabled" }

// Main conversation thread passes it from options:
thinkingConfig: X.options.thinkingConfig
```

### Decision tree

```
Is thinking globally disabled? (CLAUDE_CODE_DISABLE_THINKING=1)
  └─ YES → no thinking config sent
  └─ NO  → Does model support thinking? (j3q)
              └─ NO  → no thinking config sent
              └─ YES → Get effort budget (J3q)
                         │
                         ├─ CLAUDE_CODE_DISABLE_ADAPTIVE_THINKING=1?
                         │    └─ YES → { type: "enabled", budget_tokens: N }
                         │
                         ├─ Model supports adaptive? (Yk1)
                         │    └─ NO  → { type: "enabled", budget_tokens: N }
                         │
                         ├─ Effort budget is non-null?
                         │    └─ YES → { type: "enabled", budget_tokens: effortBudget }
                         │
                         └─ DEFAULT → { type: "adaptive" }   ← this is what v2.1.63 sends
```

## Effort budget function (J3q)

```javascript
// Raw minified
J3q(A){
  if(!A.toLowerCase().includes("opus-4-6"))return null;
  let K=jA("tengu_crystal_beam",null);
  if(K&&K.budgetTokens>0)return K.budgetTokens;
  return null
}
```

Deobfuscated:

```javascript
function getEffortBudget(model) {
    // Only applies to Opus 4.6
    if (!model.toLowerCase().includes("opus-4-6")) return null;
    
    // Check server-side feature flag (GrowthBook experiment)
    let experiment = getFeatureFlag("tengu_crystal_beam", null);
    if (experiment && experiment.budgetTokens > 0) return experiment.budgetTokens;
    
    // Default: null — no effort-based budget override
    return null;
}
```

**Key insight:** In v2.1.63, the effort budget function only checks a server-side feature flag (`tengu_crystal_beam`). It does **not** check `effortLevel` from settings.json or the `/effort` command — those don't exist yet in this version. The function returns `null` by default, which satisfies the `effortBudget === null` condition in the decision tree, routing to `{ type: "adaptive" }`.

This means the effort level concept (low/medium/high) was added later (v2.1.68+) and the budget function was updated to incorporate it. In v2.1.63, the only way to influence thinking behavior is:

1. `CLAUDE_CODE_DISABLE_ADAPTIVE_THINKING=1` → forces fixed budget
2. `CLAUDE_CODE_DISABLE_THINKING=1` → disables thinking entirely
3. Server-side `tengu_crystal_beam` experiment → sets a specific budget (controlled by Anthropic, not the user)

## The `countTokens` path

The `thinking:{type:"enabled",budget_tokens:1024}` literal is only used in `countTokens` calls (for estimating token usage), not for actual conversation. This is because the `countTokens` API endpoint requires a concrete budget — it doesn't accept `{ type: "adaptive" }`:

```javascript
// In _p6 (countTokens wrapper):
var zk8 = 1024;  // budget_tokens constant for countTokens only
var cJ4 = 2048;  // max_tokens for countTokens

// Only adds thinking if the conversation contains thinking blocks:
...z ? { thinking: { type: "enabled", budget_tokens: zk8 } } : {}
```

This is irrelevant to actual model behavior — it's just for token counting estimates.

## Subagent thinking config

All subagents (compaction, web search, agent creation, verification, side questions, btw-questions) hardcode `thinkingConfig: { type: "disabled" }`. Only the main conversation thread uses adaptive thinking.

```javascript
// Compaction:
thinkingConfig: { type: "disabled" }

// Web search:
thinkingConfig: H ? { type: "disabled" } : q.options.thinkingConfig

// Agent creation:
thinkingConfig: { type: "disabled" }

// Side questions (/btw):
thinkingConfig: { type: "disabled" }
```

The web search path is interesting — it conditionally passes through the main thinkingConfig, suggesting some tool calls may inherit the adaptive config.

## Version comparison implications

| Feature | v2.1.63 (Feb 28) | v2.1.68+ (Mar 4+) | v2.1.72+ (Mar 10+) |
|---------|-------------------|---------------------|----------------------|
| Adaptive thinking | Sent by default | Sent by default | Sent by default |
| `CLAUDE_CODE_DISABLE_ADAPTIVE_THINKING` | Works | Works | Works |
| Effort budget function | Checks feature flag only | Checks effort level | Checks effort level |
| Default effort | N/A (no effort concept) | Medium (newly added) | Medium |
| `/effort` command | Not available | Not available | Available |
| `effortLevel` in settings.json | Not available | Not available | Available (v2.1.76) |
| Output efficiency prompt | Not present | Present (v2.1.64+) | Present |

**Downgrading to v2.1.63 does NOT escape adaptive thinking.** The only benefits are: (1) no Output efficiency prompt, (2) no medium effort default (effort concept doesn't exist yet, so the server-side default applies — which was high pre-v2.1.68). The cost is losing all configuration controls (`/effort`, `effortLevel`, `showThinkingSummaries`) and months of security patches.

## Methodology notes

- Source obtained via `npm pack @anthropic-ai/claude-code@2.1.63` (SHA: `fc4103d9e1041365c0a081273ea9af1ba53f979b`)
- Analysis performed on the minified `cli.js` bundle (12,829 lines)
- Variable names deobfuscated by cross-referencing usage patterns, string literals, and API parameter names
- Function signatures inferred from their call sites and return value usage
- No runtime testing was performed — all conclusions are from static analysis
- SDK documentation strings embedded in cli.js were excluded from analysis (they contain `adaptive` references that are user-facing docs, not code logic)
